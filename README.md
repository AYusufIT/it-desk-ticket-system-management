 <h1> It Ticketing System Lifecycle (Spiceworks) examples </h1>
</p>
This project demonstrates practical hands-on experience with an IT ticketing system using Spiceworks Cloud Help Desk.

It covers the full lifecycle of IT support tickets including ticket creation, categorization, troubleshooting workflow, proper ticket closure, documentation, and escalation procedures.<br />

<h2>Objectives</h2>
  
- Create simulated user incidents
- Document ticket lifecycle
- Practice escalation procedures
- Apply closure verification standards

<h2> Skills Demonstrated</h2>

- IT Help Desk Operations
- Incident Documentation
- Troubleshooting Workflow
- Escalation Procedures
- Customer Support Process


<h2> Simulated Ticket Creation</h2>

In this example, a user submits a ticket because their monitor is not working correctly.
The issue is affecting productivity, so the ticket needs to be documented clearly with all relevant troubleshooting information.

The first step is opening the Spiceworks ticketing system and selecting New Ticket.
This simulates how an end user or support desk creates a support request.

<img src="https://i.imgur.com/bE5lCGt.png" height="50%" width="50%" alt=""/>

The requester’s information is entered along with a short subject line summarizing the issue.
A detailed description is then added explaining:

What device is affected
What the issue is
What troubleshooting has already been attempted
How to contact the user

This improves communication and reduces unnecessary back-and-forth.

 <img src="https://i.imgur.com/QdY44wn.png" height="50%" width="50%" alt=""/>

Priority is selected based on urgency and business impact.
For example:

Low = Minor inconvenience
Medium = User productivity issue
High = Department issue
Critical = Business-wide outage

This helps IT teams manage workloads effectively.

 <img src="https://i.imgur.com/aObnP0W.png" height="50%" width="50%" alt=""/>

After all relevant information is entered, the ticket is submitted.
Once created, technicians can review the ticket, track status, assign it, and begin troubleshooting.


 <img src="https://i.imgur.com/n05twmP.png" height="50%" width="50%" alt=""/>

<h2> Ticket Closure Checklist  </h2>

In this example, I worked with a Wi-Fi connectivity issue where a user’s laptop no longer connected automatically to office Wi-Fi.
Instead of focusing only on the technical fix, this lab focused on the full ticket closure process and proper documentation.

I first created a realistic support ticket based on the user’s issue:

Example issue:
“Wi-Fi no longer connects automatically on work laptop.”

What I included:

- User/requestor name
- Device or hostname
- Issue summary
- Detailed description of the problem
- Availability/contact information
- Category (Network)
- Priority level


 <img src="https://i.imgur.com/ht7gFdA.png" height="50%" width="50%" alt=""/>


After opening the ticket, I role-played the troubleshooting process.

Example fix:
- Identified expired Wi-Fi certificate
- Renewed certificate / reactivated network access
- Verified connectivity was restored

Before closing the ticket, I confirmed the issue was actually fixed from the user’s side.

Example response:

“Your Wi-Fi certificate was renewed. Please confirm that your laptop is connecting properly now.”

Why this matters:
Even if the technical fix seems complete, user confirmation ensures the problem is fully resolved.



 <img src="https://i.imgur.com/ym6ha3O.png" height="50%" width="50%" alt=""/>

I documented the root cause and actions taken.

Internal note example:
Wi-Fi certificate expired
Renewed certificate in backend
Reconnected device successfully

Purpose:
Internal notes help future technicians solve repeat issues faster.



 <img src="https://i.imgur.com/ym6ha3O.png" height="50%" width="50%" alt=""/> 

Once:

Issue was resolved
User confirmed success
Documentation was complete

<h2> Escalation Practice </h2>


In this example, a user could not access the financial database required for their job.
Basic troubleshooting was completed, including credential reset attempts and network checks, but the issue required additional permissions or backend access from another team.

I first created a realistic support ticket based on the user’s access issue.

Example issue:
“User cannot access financial database.”

What I included:
User/requestor information
Device or account details
Issue summary
Description of what system or database access was missing
Troubleshooting already attempted
Category (Software / Access Request)
Priority level based on business impact


 <img src="https://i.imgur.com/zk42FFF.png" height="50%" width="50%" alt=""/> 


Before escalating, I completed standard Level 1 troubleshooting such as checking login credentials and verifying network connectivity to confirm the issue could not be resolved through basic support steps.

Since the issue involved database permissions and access beyond normal help desk scope, I determined the ticket needed to be escalated to a more advanced team.

I reassigned the ticket to the correct department or higher-level technician responsible for database or access management.

 <img src="https://i.imgur.com/rHkVHxi.png" height="50%" width="50%" alt=""/> 


I updated the ticket with all relevant user and account information, including what access was needed, what troubleshooting had already been completed, and why the issue required escalation. This ensured the next team had the full context needed to continue efficiently.

I used a public response to inform both the user and the assigned team that the ticket was being escalated. This kept the user informed about next steps while ensuring the receiving team had clear visibility into the request and could take over without delays.
